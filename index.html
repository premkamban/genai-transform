RAG Concepts

  • Embeddings: float vectors encoding semantic meaning
  • Cosine similarity: measures meaning, not keyword overlap
  • Chunking: fixed (512 chars), overlap (128), semantic split
  • Dense vs sparse retrieval: embeddings vs BM25 keyword
  • HyDE: embed a hypothetical answer, not the raw question
  • MultiQueryRetriever: 3 rephrasings → 3 retrievals → merge
  • RAGAS: faithfulness, answer_relevancy, context_precision
  • Token budget: system prompt + history + tool results + retrieved chunks all compete for the same fixed context window — you allocate it, the model doesn't

Context Engineering — RAG Is One Tactic, Not the Whole Job

  • Context engineering = everything that ends up in the model's context window, not just retrieved docs
  • Conversation memory: raw buffer vs rolling summary vs vector-store memory — each trades cost against recall differently
  • Tool-output injection: how you format a function result back into context shapes how well the next model call uses it
  • Context ordering: models weight recent and start-of-context tokens differently — placement matters, not just presence
  • RAG is the retrieval half of this discipline. This month you also build the memory half

  agent { docker { image 'python:3.11-slim' } }
 
  environment {

    stage('Lint + Test') {

        sh 'pytest tests/ -v --tb=short'
      }
    }
    stage('AI Eval Gate') {
      // Fail pipeline if RAG quality drops
      steps {
        sh 'pip install deepeval && deepeval test run tests/eval/'
      }
    }
    stage('Docker Build + Push to ECR') {
      steps {
        script {
          // Auth with ECR
          sh 'aws ecr get-login-password --region $AWS_REGION | docker login --username AWS --password-stdin $ECR_REPO'
          sh 'docker build -t $ECR_REPO:$IMAGE_TAG .'
          sh 'docker push $ECR_REPO:$IMAGE_TAG'
        }
      }
    }
    stage('Deploy to EKS') {
      steps {
        // Update kubeconfig for EKS cluster
        sh 'aws eks update-kubeconfig --name my-cluster --region $AWS_REGION'
        // Helm upgrade with new image tag
        sh 'helm upgrade --install ai-app ./helm/ai-app --set image.tag=$IMAGE_TAG --set image.repository=$ECR_REPO'

        sh 'kubectl rollout status deployment/ai-api --timeout=300s'
      }
    }
  }
  post {
    failure {
      sh 'kubectl rollout undo deployment/ai-api'
      slackSend channel: '#alerts', message: 'Deploy FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}'
    }
  }
}

📬AWS SQS
Amazon SQS — Serverless Queue, Zero Ops, AWS-Native
Best for AI app async jobs on AWS · Lambda triggers · $0.40 per million messages · no servers to manage
+
Core Concepts You Must Know
Standard vs FIFO Queue
Standard: At-least-once delivery, messages may arrive out of order. Fine for most AI tasks (re-embedding a doc doesn't care about order). FIFO: Exactly-once, strictly ordered. Use when order matters (process chapter 1 before chapter 2).
Visibility Timeout
When a worker picks up a message, it becomes invisible to other workers for N seconds (your visibility_timeout). If your Lambda takes 30s to embed a document, set timeout to 60s. If the worker crashes, the message reappears and another worker picks it up.
Dead Letter Queue (DLQ)
If a message fails 3 times (embedding fails because the PDF is corrupt), move it to a DLQ instead of blocking the queue. Inspect failed messages without losing them. Set maxReceiveCount=3 to trigger DLQ move.
CLI Commands
# Create a standard queue
aws sqs create-queue \
  --queue-name ai-doc-ingestion \
  --attributes VisibilityTimeout=300,MessageRetentionPeriod=86400
 
# Send a job (e.g. "ingest this PDF")
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123/ai-doc-ingestion \
  --message-body '{"s3_key": "docs/report.pdf", "user_id": "u123"}'
 
# Receive + process (long-poll, waits up to 20s for message)
aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123/ai-doc-ingestion \
  --wait-time-seconds 20 \
  --max-number-of-messages 10
 
# Delete message after successful processing
aws sqs delete-message \
  --queue-url ... \
  --receipt-handle <handle-from-receive>

🌊Apache Kafka
Kafka — Event Streaming at Scale, Message Replay, Real-Time Pipelines
GE-scale event streaming · millions of events/sec · messages retained days/weeks · multiple consumers can replay same events
+
Core Concepts
Topic / Partition / Offset
Topic = named stream (e.g. rag.document.uploaded). Partition = parallel lanes within a topic for throughput. Offset = position in the log — consumers track their own offset so they can replay from any point.
Consumer Group
Multiple consumers in the same group share partitions — scales processing. Different groups (e.g. RAG embedder + audit logger) each get ALL messages independently. SQS can't do this multi-consumer pattern.
Message Retention (the key difference vs SQS)
Kafka keeps messages for 7 days by default (configurable forever). You can replay all document upload events to re-index your entire RAG system. SQS deletes messages after processing. Kafka doesn't.
Python Producer + Consumer

  's3_key': 'docs/report.pdf',

  'rag.document.uploaded',

📨ActiveMQ / RabbitMQ
ActiveMQ / RabbitMQ — AMQP Brokers, Complex Routing, NestJS Native
Battle-tested message brokers · AMQP protocol · NestJS microservices module has built-in support · GE enterprise Java shops use ActiveMQ

Core Concepts
Exchange → Binding → Queue
Producer sends to an Exchange. The exchange routes to Queues based on routing rules (Bindings). Three types: Direct (exact match), Topic (wildcard like rag.#), Fanout (broadcast to all queues). More expressive than SQS.

🐘PgBoss
PgBoss — Job Queue Inside Postgres, Zero New Infrastructure
Jobs stored in a Postgres table · perfect when you already have RDS · no separate queue service · Node.js / TypeScript native · transactional job creation
+
Why PgBoss Is Underrated for AI Apps
Jobs in a Postgres Table
PgBoss creates a pgboss.job table in your existing Postgres database. A job is just a row. You can query it with SQL, join it with your business data, and inspect it with your existing DB tools — no separate monitoring dashboard needed.
Transactional Job Creation
Create a user AND enqueue an embedding job in the same Postgres transaction. If the user creation fails, the job is never created. Impossible with SQS or RabbitMQ — you can't atomically write to a DB and send a message in one transaction.
Scheduled + Delayed Jobs
Schedule a re-indexing job to run every Sunday at 2am. Delay an email by 30 minutes. Set a job to run in 7 days. All built-in. All in Postgres. No cron infrastructure.
TypeScript / NestJS Example

// The killer feature: atomic job + DB write
// User signs up → embed their profile simultaneously
// If user write fails → job never created (no orphan jobs)
async function createUserAndQueueEmbedding(userData) {
  return await db.transaction(async (trx) => {
    // Step 1: create user in transaction
    const user = await trx('users').insert(userData)
    // Step 2: enqueue embedding job in same transaction
    await boss.send('embed-user-profile', {
      userId: user.id,
      profile: userData.bio
    }, { db: trx }) // ← pass the transaction!
    // If ANYTHING throws, BOTH roll back

  })
}

Batter Fermentation Science

  • Classic idli/dosa ratio: 4:1 rice to urad dal (adjust toward 3:1 for softer idli, 4:1 for crisper dosa)
  • Soak rice and dal separately (4–6 hrs), grind dal to a light, aerated paste — air incorporation matters more than fineness
  • Fermentation is wild yeast + lactic acid bacteria — warm climates (25–30°C) ferment in 8–10 hrs, colder climates need an oven-light trick or longer time
  • Salt after fermentation, not before — salt slows the microbial activity

The Kuzhambu / Rasam / Kootu Family

  • Rasam — thin, tamarind-tomato-pepper base, no dal thickening (milagu rasam, paruppu rasam, tomato rasam)

  • Kuzhambu — tangier, oil-forward gravies (vatha kuzhambu, kara kuzhambu, mor kuzhambu)

  • Meat marination with fresh-ground masala, not pre-mixed powder, is what separates real Chettinad from imitation
  • Heavy use of shallots (sambar onions) instead of regular onions — sweeter, faster-cooking base

Regional Style Notes

  • Kongunadu (Coimbatore belt): earthier, less oil, arisi paruppu sadham, kaara kuzhambu with peanuts
  • Madurai: bold, mutton-forward — Madurai mutton curry, kari dosai, paruthi paal (cottonseed milk dessert)
  • Thanjavur: temple-influenced vegetarian cooking — subtler spicing, ghee-forward, vella sadam (sweet rice)

Plating & Presentation

  • Banana-leaf service order: salt/pickle first (top-left), then in a fixed traditional sequence — learn it, then modernise deliberately
  • Height, colour contrast, and negative space on a plate — apply restrained modern plating to traditional dishes
  • Garnish with function, not decoration: a curry leaf tempering on top is flavour, not just colour

Formal Training Options

  • NCHMCT / IHM networkDiploma/DegreeIndia's national hotel management & catering body
  • WorldChefs (World Association of Chefs Societies)CertificationGlobal culinary standards & competitions

Menu Design & Costing

  • Cost every dish per portion: raw ingredient cost ÷ target food-cost % = menu price
  • Balance a menu across prep-heavy and quick-fire dishes so the kitchen doesn't bottleneck at service
  • Batch-cook bases (podis, masalas, tempering oils) ahead — scale technique without scaling chaos

Catering & Scale

  • Scaling a fermented batter recipe isn't linear — test at 2×, 5×, and 10× before trusting a large batch
  • Hold temperatures for sambar/rasam differ at catering scale — plan for extended service windows
  • Practice mise en place for a 50-cover event, not just a 4-person meal
